Etapa 4 — Parametrização de benefícios

Tipos, regras, valores e fornecedores · ETA: 20–30 min

URL base: #/wizard/etapa-4/ · Código: frontend/src/pages/wizard/stage4.tsx

Objetivo: criar a “receita” de cada benefício (tipo, nome interno, rateio, valores, fornecedor, VT) sem vincular colaboradores — o vínculo por funcionário vem após o eSocial. Depende das etapas 1–3 validadas no fluxo.

passo-1 — Entrada

Texto orientativo e alertas de dependência. Sem campos persistidos.

passo-2 — Tipos de benefícios

Tabela e cards por tipo fixo do sistema. Colunas: Benefício, Status (Configurado / Não configurado).

Valor type (API) Rótulo na UI
MEAL_VOUCHER Vale refeição
FOOD_VOUCHER Vale alimentação
TRANSPORT_VOUCHER Vale transporte
HEALTH_PLAN Plano de saúde
LIFE_INSURANCE Seguro de vida
OTHER Outro

passo-3 — Cadastro do tipo (novo benefício)

Cria registro com regra inicial 100% empresa e sem dependentes; valores iniciais editáveis nos passos seguintes.

Campo Obrig. Para que serve Como funciona
type Sim Categoria do benefício. Mesmos valores do passo 2; se OTHER, alerta para usar nome interno descritivo.
internalName Sim Nome exibido na folha e relatórios. Ex.: “Vale refeição — padrão mensal”.
valueType Sim Como o valor se aplica. DAILY (diário) ou MONTHLY_FIXED (mensal fixo).
defaultValue Não Valor padrão em R$. Número; enviado como 0 se vazio no create inicial.

Payload inicial inclui paymentRuleType: COMPANY_100, understoodAck: true, hasDependents: false (ajustáveis no passo 4).

passo-4 — Regras de cálculo

Campo Obrig. Para que serve Como funciona
benefitId Sim Qual benefício será atualizado. Select pesquisável; ao mudar, preenche demais campos com valores atuais.
paymentRuleType Sim Quem arca com o titular. COMPANY_100, EMPLOYEE_100 ou SPLIT_PERCENT.
companyPercentage Se SPLIT % empresa (titular). 0–100; exibido só em rateio.
employeePercentage Se SPLIT % colaborador (titular). 0–100.
hasDependents Não Há cobertura/rateio para dependentes? Checkbox; se marcado, exige % empresa e % colaborador para dependentes.
dependentCompanyPercentage Se dependentes % empresa sobre dependentes. 0–100.
dependentEmployeePercentage Se dependentes % colaborador sobre dependentes. 0–100.

Persistência: PUT /benefits/:id com merge dos campos acima.

passo-5 — Valores

Campo Obrig. Para que serve Como funciona
benefitId Sim Benefício alvo. Select sincroniza valueType e defaultValue.
valueType Sim Diário ou mensal fixo. DAILY / MONTHLY_FIXED.
defaultValue Sim Valor padrão R$. InputNumber ≥ 0, passo 0,01.

passo-6 — Fornecedor

Campo Obrig. Para que serve Como funciona
benefitId Sim Benefício do fornecedor. POST /benefit-suppliers
name Sim Razão social ou nome fantasia do operador. —
taxId Não CNPJ. Máscara; só dígitos no envio; validado se informado.
isPrimary Não Marca fornecedor principal do benefício. Checkbox; padrão falso.
Seletor de fornecedor (layout) Não Escolher a quem anexar o arquivo de layout. Deve selecionar um fornecedor já salvo antes do upload.
Arquivo de layout Não CSV/XLS/PDF do convênio (Alelo, Caju, etc.). POST /benefit-suppliers/:id/layout-file multipart.

Tabela de fornecedores: Benefício (nome interno), Nome, CNPJ, Principal.

passo-7 — Regras específicas (VT)

Campo Obrig. Para que serve Como funciona
benefitId Sim Qual VT (tipo TRANSPORT_VOUCHER). Lista filtrada; se vazia, cadastrar VT no passo 3.
transportDiscountPercent Sim Percentual de desconto em folha (teto legal na UI). Máximo 6% (referência CLT); passo 0,5; rejeita > 6.

passo-8 — Confirmação

Reforça que não há vínculo com colaboradores nesta etapa. O submit cria um benefício exemplo se marcado.

Campo Obrig. Para que serve Como funciona
ok Sim Confirmar leitura do aviso. Checkbox “Entendi”; validação impede envio sem marcar.
name Não Nome interno opcional do benefício exemplo. Default no servidor: “Benefício exemplo” se vazio.

Ao enviar: POST /benefits com tipo VR fixo no código de exemplo, understoodAck vindo do checkbox.

resumo

  • Tipos cadastrados (contagem benefícios > 0)
  • Regras definidas
  • Valores configurados
  • Fornecedores vinculados
  • VT com passo 7 (se existir benefício transporte)
